# MCP Apps Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship MCP Apps behind a compact public MCP surface using **Variant B**: one public `open_app` entry-point, resource-driven read-side UI, and at most one additional coarse-grained `app_command` tool for mutation-heavy app workflows.

**Architecture:**
- Keep the existing app-specific Rust/service logic private and reusable.
- Replace the planned public `open_*` / `apply_*` tool explosion with a single public launcher (`open_app`) and session-backed resources.
- Add MCP `resources` capability so app views are read through `resources/read` using `ui://memory/app/...` URIs.
- Defer `app_command` until needed for APP-03 / APP-04; reuse canonical memory tools (`invalidate`, `resolve`, `ingest`, etc.) wherever the business intent already matches.

**Tech Stack:** Rust 2024, `rmcp` 1.3.0, Tokio, SurrealDB 3.x, serde, schemars

---

## Revision Summary — 2026-03-31

This plan supersedes the earlier public-API design that exposed a separate MCP tool for each app screen and action.

### Why the plan changed

- MCP tools are model-discoverable and should remain small in number.
- MCP resources are the correct substrate for app read-side views.
- The repository now intentionally enforces a minimal public tool surface.
- The existing codebase already contains substantial private app groundwork, so the cheapest path is to **wrap and unify** it rather than throw it away.

### Final target surface

**Public tools:**
- Canonical six memory tools
- `open_app`
- `app_command`

**Public resources:**
- Session-backed URIs under `ui://memory/app/{app}/{session_id}`

**Private/internal only:**
- `open_memory_inspector`
- `refresh_memory_inspector`
- `open_temporal_diff`
- `export_temporal_diff`
- `open_memory_inspector_from_diff`
- `open_ingestion_review`

## Design Principles for This Revision

1. **One honest entry-point beats hidden magic.**
   `open_app` is public and discoverable. App-specific openers remain private.

2. **Read-side belongs to resources.**
   UI state and screen payloads are read via `resources/read`, not by publishing more tools.

3. **Reuse business-intent tools where possible.**
   Example: invalidation from Inspector should continue to use the canonical public `invalidate` tool when feasible.

4. **Add `app_command` only when needed.**
   APP-03 and APP-04 are the main candidates; read-oriented apps should not force this tool into existence prematurely.

5. **Preserve existing private code while pivoting the public contract.**
   The legacy `open_*` handlers are implementation details during migration.
